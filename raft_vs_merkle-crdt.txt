With Raft:
  1. 50 peers editing a doc → Raft elects leader among them
  2. Leader collects all CRDT ops, batches them
  3. Leader encrypts batch, submits to JAM
  4. Leader goes offline → election timeout (seconds), new leader elected
  5. During election: ops buffer locally, nothing goes to JAM
  6. New peer joins → must discover the current leader

  With Merkle-CRDTs:
  1. 50 peers editing a doc → each records ops in local DAG
  2. Peers gossip root CIDs via PubSub (or any channel)
  3. Any peer receiving a new root CID fetches missing nodes from whoever has them (other peers OR JAM DA
  layer)
  4. Any peer can push their root to JAM at any time (e.g., periodically, or on every op)
  5. Peer goes offline → nothing happens, others continue, offline peer syncs when back
  6. New peer joins → fetches current root CID from JAM, walks the DAG, done
